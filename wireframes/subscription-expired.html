Note tecniche di implementazione

Decisioni architetturali e flusso di gestione abbonamento scaduto

Approccio consigliato: Scenario 1 (Dashboard Suspended)

Piu' professionale del semplice redirect al login. L'utente capisce che:

  • Il suo account esiste ancora
  • I dati sono al sicuro
  • Sa esattamente cosa fare per risolvere
  • Non pensa che sia un errore tecnico

Implementazione TenantMiddleware

Invece di fare logout, il middleware controlla l'abbonamento e:

  • Se scaduto: redirect a /dashboard/suspended
  • La rotta /dashboard/suspended e' l'unica accessibile
  • Tutte le altre rotte /dashboard/* rimandano li'
  • Il logout resta funzionante

Implementazione pagine pubbliche

BookingController e MenuPageController gia' controllano is_active del tenant. Aggiungere il check subscription nella stessa logica:

  • Se tenant attivo MA abbonamento scaduto: mostra pagina sospensione
  • Se tenant disattivato: 404 come prima
  • La pagina pubblica mostra i contatti del ristorante (phone, email, address dal tenant)
  • Messaggio neutro, nessun riferimento a "abbonamento"

API Endpoints

  • GET /api/v1/tenants/{slug}/availability → JSON {"error": "service_suspended", "message": "..."} con status 403
  • GET /api/v1/tenants/{slug}/menu → stessa risposta
  • Il widget JS deve gestire il caso 403 e mostrare il messaggio embed (Scenario 3)

File coinvolti

  • app/Middleware/TenantMiddleware.php — check subscription, redirect a /dashboard/suspended
  • app/Controllers/Dashboard/SuspendedController.php — NUOVO, mostra la pagina sospensione
  • views/dashboard/suspended.php — NUOVO, pagina sospensione dashboard
  • views/booking/suspended.php — NUOVO, pagina pubblica sospensione
  • app/Controllers/BookingController.php — MODIFICA, check subscription
  • app/Controllers/Menu/MenuPageController.php — MODIFICA, check subscription
  • app/Controllers/Api/* — MODIFICA, return 403 JSON
  • public/assets/js/booking-widget.js — MODIFICA, gestire 403
  • config/routes.php — MODIFICA, aggiungere rotta /dashboard/suspended
