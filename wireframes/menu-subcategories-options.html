Opzione A — Albero gerarchico
+ Massima flessibilità — nesting illimitato (Antipasti → Di Mare → Crudi → ...)
− Overkill — i menu ristorante raramente vanno oltre 2 livelli
− Complessità alta — query ricorsive, sort_order multi-livello, UI drag&drop nested
− UX confusa — il ristoratore può creare gerarchie troppo profonde, l'utente si perde

Opzione B — Sottocategorie a 1 livello ★ Consigliata
+ Copre il 99% dei casi — Antipasti → Di Mare / Di Terra è il massimo necessario
+ UI pulita — indentazione chiara, bottone "+ Sottocategoria" per ogni parent
+ Sticky nav pulita — mostra solo le categorie principali
− 1 colonna + 1 check — migration `parent_id` + enforce depth = 1 nel controller
+ Pattern standard — è come funzionano TheFork, Deliveroo, Glovo

Opzione C — Sezioni/divisori dentro la categoria
+ Semplicissima da implementare — 1 colonna VARCHAR su menu_items, nessuna migration complessa
+ Zero nuova UI — solo un campo testo in più nel form piatto
− Raggruppamento "soft" — la sezione è una stringa libera (errori typo: "di mare" vs "Di Mare")
− Nessun CRUD dedicato — non puoi rinominare una sezione in blocco, devi modificare ogni piatto
− Meno strutturato — il ristoratore deve ricordare i nomi esatti delle sezioni
