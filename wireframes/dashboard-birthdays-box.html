3 Note tecniche d'implementazione

Query SQL

  • Database: la colonna customers.birthday esiste già (nessuna migration)
  • Filtro: clienti del tenant con birthday IS NOT NULL e compleanno nei prossimi 30 giorni
  • Cross-year: gestire il caso dicembre→gennaio (es. nato 28-12-1985, oggi 5-12-2026 → tra 23 gg)
  • SQL pseudo: WHERE DATE_FORMAT(birthday, '%m-%d') BETWEEN ... AND ... oppure calcolato a livello app per più chiarezza
  • Query NON filtra su marketing_consent: è uso operativo interno, non broadcast email. Il ristoratore può chiamare anche chi non ha dato consenso marketing

Backend

  • app/Models/Customer.php: nuovo metodo findUpcomingBirthdays(int $tenantId, int $daysAhead = 30): array
  • Ritorna: id, first_name, last_name, birthday, phone, email, total_bookings, segment (calcolato)
  • Calcolo "tra X giorni": preferibilmente in PHP per chiarezza, no SQL DATEDIFF complicate
  • Limit 10-20 in pagina dashboard, link "Vedi tutti" per pagina dedicata
  • Ordine: prima il compleanno più vicino

Frontend

  • View: views/dashboard/home.php — aggiungere nuovo card box dove scelto (sidebar destra o left column)
  • CSS: nuove classi .bd-* in dashboard.css
  • Mobile: collassa in lista 1-col, azioni inline restano, chip data più piccolo
  • Action buttons: link tel:, https://wa.me/ (numero pulito), profilo cliente
  • Empty state: card resta visibile con messaggio esplicativo (non scompare)

Decisioni di scope

  • Solo visualizzazione: nessuna campagna automatica auguri da questo widget. È un trampolino per l'azione manuale.
  • No filtro marketing_consent: uso operativo interno (non sta mandando broadcast). Anche clienti che hanno revocato consenso vengono mostrati per consentire chiamata diretta.
  • Limite default 30 giorni: configurabile in futuro se richiesto, ma 30 è il giusto equilibrio.
  • Posizione: vedi punto 4 sotto.
4 Mia raccomandazione

Variante A (sidebar compatta)

  • Perché A: la dashboard ha già molti box centrali (Prossimi in arrivo, Riepilogo servizi, Confronto settimana, No-show). Mettere "Compleanni" nella colonna sinistra rende la pagina più affollata. La sidebar destra ha già "Quick actions" — il box "Compleanni" si innesta naturalmente lì.
  • Trade-off accettato: niente azioni rapide inline. Si clicca sulla riga → profilo cliente CRM dove ci sono telefono, email, WhatsApp, storico, note. Non perdi nulla, fai 1 click in più.
  • Densità informativa: 5-7 righe visibili senza scroll = giusto compromesso tra visibilità e ingombro.
  • Tempo implementazione: ~2.5h (query, view, CSS, link al profilo).
